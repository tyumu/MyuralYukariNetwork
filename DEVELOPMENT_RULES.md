# Development Rules

このリポジトリで今後の開発を進めるときの基準をまとめます。

## 必須

- 変更は最小単位で行う。
- 置き換えたコードや不要になったファイルは、その場で削除する。
- `memU/` 配下は、明示的な依頼があるまで編集しない。
- 既存の命名、構成、責務分割を勝手に広げない。
- 1 つの関数やファイルに処理を詰め込みすぎない。

## Go バックエンド

- `internal/api` 配下は、エントリーポイント (`chat_handler.go`, `status_handler.go`) とフロー (`chat_flow.go`) の責務分離を維持する。
- 入力検証、検索、生成、保存、レスポンス組み立ては別関数に分ける。
- 設定値は `internal/config` に集約し、ハードコードを増やさない。
- エラーは隠さず、呼び出し元に意味が分かる形で返す。

## フロントエンド

- UI 状態は永続化の要否を意識して持つ。
- 古い入口ファイルや未使用コンポーネントは残さない。
- API ベース URL や環境差分はコードに直書きしない。

## ドキュメント

- 実装と説明がずれたら、まず説明を直す。
- 起動手順、ポート、環境変数は実体と一致させる。
- 使われていない前提ファイルがあれば削除するか、明示的に非推奨化する。

## 確認

- Go の変更後は `go test ./...` を確認する。
- 影響があるフロントエンド変更後はビルドか実行確認を行う。
- ルールを壊す大きな変更をする場合は、先に理由を文書化する。

## 禁止

- `memU/` の無断変更。
- 不要になったコードやファイルを残したまま進めること。
- 構成の説明だけ更新して、実装の整理をしないこと。
